Useful Git Information

Skip to end of metadata
  • Created by Scott Leon, last modified by Yury Kartsev on Jun 10, 2015
Go to start of metadata

If you find the information on this page useful, thank The API Team for creating the page (copied from Useful Git Information)

 

Use with caution and make sure you understand the underlying concepts of git. Before you dive into your day to day work, please read the git-scm.com getting started guide, then read first 3 sections of this doc, then watch 2 tutorial videos located here (TODO move pages to github wiki) as an overview of the phoenix-services git process. Note: after you watch videos, remember to NOT use squash for merging branches with more than 1 contributor. Details about why not to use squash and lot of other useful advices can be found in this powerpoint doc which was created based on phoenix-services git experience.

Awesome visual cheatsheet

This does a great job of outlining handy commands in a visual way: http://www.ndpsoftware.com/git-cheatsheet.html

Overall git process

  • Pull from remote master at least daily.
  • Always create a new branch for your functionality with your initials plus a feature/fix name (or bug number). For example : mm_myawesomefix, sg_PHX_3456.
  • Rebase your local branch from remote master at least daily.
  • Push your local branch to the remote origin as often as you need to.
  • When done with your development, push your local branch to the remote origin and ask for a pull request via the web interface.
  • Wait (and possibly goad/bribe/plead) for +1 from a team member!
  • Then go through merge process to master. Either automatic (if branch was recently rebased - highly recommended option) or manual.
  • When PR for a release branch is merged (or for any other branch which will not be merged into master), developer is responsible forcherry-picking particular commit (or set of commits) to master branch (or any other parent branch). Preserving same comment while cherry-picking is a must! Comments may be prepended with an "[CP]" to indicate that it was a cherry-pick.
  • Builds are (usually) done off master automatically. Check Jenkins to make sure your changes resulted in a successful build.

When should I do a rebase vs. merge?

This StackExchange thread has a really good guideline for when to rebase vs. merge: http://stackoverflow.com/questions/2472254/when-should-i-use-git-pull-rebase

In summary: Rebase when you're doing development that isn't drastically diverging from master (and you can rebase early and often). Mergewhen your development is a large, new chunk of functionality that will diverge from master.

This was from a great article (forgot the source):

If you treat rebasing as a way to clean up and work with commits before you push them, and if you only rebase commits that have never been available publicly, then you’ll be fine. If you rebase commits that have already been pushed publicly, and people may have based work on those commits, then you may be in for some frustrating trouble.

Useful commands

Here's some git commands you might need to reference in your day to day work with phoenix-services

Make sure you understand what these do and the git process before using them


Git autocompletion script

In order for autocompletion of git commands (and branch names) to work, you need to execute the following 2 commands:
    1. curl https://raw.github.com/git/git/master/contrib/completion/git-completion.bash -o ~/.git-completion.bash (or get from attachments of this page: git-completion.bash)
    2. echo "source ~/.git-completion.bash" >> ~/.bash_profile

Setup / Config

    • git config color.ui true -> shows git output in color
    • git config format.pretty format:'%Cred%h%Creset -%C(yellow)%d%Creset %s - %Cgreen%cd (%cr) %C(cyan)<%an>%Creset' -> sets default pretty format (instead of always using --pretty=format:...) for git log to be one informative colored line: 
    • git config log.date local -> sets option to display local time in the output
    • git status -> Display the current git state.
    • git log --since=yesterday --author=Gilbane -> Display log of changes since yesterday for author.

Branches

    • git checkout master -> Switch your local branch to the master branch.
    • git branch newbranchname -> Create a local branch called newbranchname.
    • git checkout -b newbranch -> Create and checkout on a new branch in one operation.
      • Branch names should start with your initials, and then describe the feature using underscores.
    • git branch -a -> Show all branches, remote and local.
    • git checkout newbranchname -> Switch to the named branch newbranchname.
    • for k in `git branch|perl -pe s/^..//`;do echo -e `git show --pretty=format:"%Cgreen%ci %Cblue%cr%Creset" $k|head -n 1`\\t$k;done|sort –r-> Show when all branches were created
    • Show all branches that contain phx in their name sorted in reverse date order of their last update. This is a useful way to look at older branches that might be removed.

for k in `git branch -r | grep phx | perl -pe 's/^..(.*?)( ->.*)?$/\1/'`; do echo -e `git show --pretty=format:"%ci %cr" $k -- | head -n 1`\\t$k; done | sort

Tracking and Committing

    • git add filename -> Track a new file, or stage an existing modified file.
    • git commit -m "message" -> Commit all changes that have been added to this branch in the local repository.
    • git log -> Displays all commits to current branch starting from the most recent one
    • git diff -> Display differences between any modified files and the current branch state.
      • git diff --cached displays differences between staged changes and the current branch state.
    • git reset HEAD filename -> UNDOABLE - Reset a file to its state at the beginning of the current branch.

More git log details

The git log command has a large number of parameters. This site has some useful examples. If you want to see what's changed after a certain date and/or time use

 git log --after=10.am
 git log --after=yesterday

You can use the --before parameter to check what changed in a branch before a specified date. The date/time values are in 'approxidate' format. Any of the following are valid. Quotes are only needed when the date contains spaces.

 "29/April/2014 23:59:59"

"april 29" (Assumes this year)

"april 10 2014"

"2014-04-29 18:10:00"

2014-04-29

"3 weeks ago"

3.weeks.ago

Remote Server

    • git fetch origin -> Fetch any work that has been pushed to the server since last cloned/fetched. This will NOT auto merge!
    • git pull -> automatically merges changes from remote origin where possible on ALL branches.
    • git push origin _newbranchname_ -> Share your new branch to the remote repository. Copies all committed changes on the local branch newbranchname to the remote repo.

Merging

    • git merge --squash from_branch
      • Will merge changes from the branch to the current branch you're in.
      • Will automatically merge where possible, but you may need to do some delicate manual merging.
    • After merge: git commit -m "merging functionality summary for branch newbranchname".
    • To squash a bunch of commits into one commit in the same branch, see this helpful article:http://stackoverflow.com/questions/6934752/git-combining-multiple-commits-before-pushing

While addressing merge conflicts, if you want to keep one file or the other (use with caution), you can there’s a --theirs and --ours options on the merge command. The former will keep the version of the file that you merged in, and the other will keep the original one.

The following commands will keep the original file for test.html, and then use the merged in file only for test.java.

            git checkout --ours index.html
git checkout --theirs test.java

See this blog post for more details.

 

GUIs

    • gitk --all -> Intriguing graphical display of all branches, including log comments.
    • git gui -> Graphical display of current branch.

Adding a new project to a remote repo:

http://stackoverflow.com/questions/4658606/import-existing-source-code-to-github

  • git remote add origin [url/ssh]
  • git push origin master
  • git branch --set-upstream master origin/master [or whatever your branch is]

Doing a Git rebase:

  • Commit changes to your branch
  • Checkout master
  • Pull from master
  • Checkout your branch
  • git rebase master
  • Resolve any conflicts
  • git rebase --continue

Useful Github Info

  • Omit whitespace in a PR - Add "?w=1" to the PR link like this: "https://github.com/tmfi/phoenix-services/pull/1570/files?w=1"
